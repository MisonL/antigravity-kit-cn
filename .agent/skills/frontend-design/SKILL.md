## ⚠️ 关键：询问胜过假设 (ASK BEFORE ASSUMING) —— 强制要求

> **停止！如果老板的请求是开放性的，请不要默认使用您个人的偏好。**

### 当需求模糊时，请主动询问：

**未指定颜色？** 询问：

> “您更倾向于哪种色板？（蓝色/绿色/橙色/中性色/其他？）”

**未指定风格？** 询问：

> “您追求的是哪种风格？（极简/大胆/复古/未来感/有机线条？）”

**未指定布局？** 询问：

> “您对布局有偏好吗？（单栏/网格/非对称/全宽？）”

### ⛔ 应避免的默认倾向（反安全避风港原则）：

| AI 默认倾向                   | 弊端                  | 替代思考路径                           |
| ----------------------------- | --------------------- | -------------------------------------- |
| **Bento 网格 (现代陈词滥调)** | 充斥在每个 AI 设计中  | 为什么这些内容**必须**用网格展示？     |
| **左右分割型 Hero**           | 刻板且无趣            | 尝试超大排版或纵向叙事如何？           |
| **弥散/极光渐变**             | “新式”偷懒背景        | 尝试激进的色彩配对？                   |
| **玻璃拟态 (Glassmorphism)**  | AI 意淫中的“高级感”   | 尝试高对比度的纯色扁平化风格？         |
| **深青色 / 金融蓝**           | 躲避紫色禁令的安全岛  | 为什么不用红色、黑色或荧光绿？         |
| **“赋能 / 助力”类词汇**       | AI 生成的典型文案     | 一个真人会怎么表达这个意思？           |
| 暗色背景 + 霓虹发光           | 过度使用，满是“AI 味” | 品牌真正需要的是什么？                 |
| **全圆角处理**                | 平庸且保守            | 在哪里可以尝试锋利的、粗野主义的边缘？ |

> 🔴 **“您选择的每一个‘安全’结构，都让您离平庸的模板更近一步。请勇于冒险。”**

## 1. 约束分析 (Constraint Analysis) —— 始终放在首位

在进行任何设计工作之前，请先回答以下问题或询问老板：

| 约束项     | 问题                   | 为什么重要                  |
| ---------- | ---------------------- | --------------------------- |
| **时间线** | 有多少时间？           | 决定了设计的复杂度          |
| **内容**   | 已就绪还是使用占位图？ | 影响布局的灵活性            |
| **品牌**   | 是否有既定的准则？     | 可能规定了特定的颜色/字体   |
| **技术栈** | 使用什么技术？         | 影响实现能力 (Capabilities) |
| **受众**   | 具体是谁？             | 驱动所有的视觉决策          |

### 受众 → 设计策略 (Audience → Design Approach)

| 受众群体           | 设计思考方向                 |
| ------------------ | ---------------------------- |
| **Z 世代 (Gen Z)** | 大胆、快速、移动优先、真实感 |
| **千禧一代**       | 整洁、极简、价值驱动         |
| **X 世代**         | 熟悉感、值得信赖、逻辑清晰   |
| **婴儿潮一代**     | 易读、高对比度、操作简单     |
| **B2B / 企业级**   | 专业、数据导向、建立信任     |
| **奢侈品/高端**    | 克制的优雅、大量留白         |

## 2. UX 心理学原则 (UX Psychology Principles)

### 核心法则 (务必内化)

| 法则                               | 原则                    | 应用指南                       |
| ---------------------------------- | ----------------------- | ------------------------------ |
| **希克定律 (Hick's Law)**          | 选项越多 = 决策越慢     | 减少选项，使用渐进式披露       |
| **菲茨定律 (Fitts' Law)**          | 越大 + 越近 = 越易点击  | 设定合理的 CTA 点击域大小      |
| **米勒定律 (Miller's Law)**        | 工作记忆仅能容纳约 7 项 | 对内容进行分块 (Chunking) 处理 |
| **孤立点效应 (Von Restorff)**      | 差异化 = 易记忆         | 让 CTA 按钮在视觉上保持独特    |
| **系列位置效应 (Serial Position)** | 首尾的内容最易被记住    | 将关键信息放在开头或结尾       |

### 情感设计层级 (Emotional Design Levels)

```
本能层 (Visceral) (瞬时) → 第一印象：色彩、意象、整体氛围
行为层 (Behavioral) (使用) → 使用过程：速度、反馈、效率
反思层 (Reflective) (记忆) → 事后： “我喜欢这个设计所代表的我”
```

### 信任建立 (Trust Building)

- 在敏感操作上提供安全指示。
- 在相关位置展示社会认同 (Social proof)。
- 提供清晰的联系方式/支持入口。
- 保持一致且专业的设计。
- 透明的政策说明。

## 3. 布局原则 (Layout Principles)

### 黄金分割率 (φ = 1.618)

```
用于比例和谐：
├── 内容 : 侧边栏 = 约为 62% : 38%
├── 各级标题大小 = 前一级 × 1.618 (实现戏剧性的比例)
├── 间距可遵循：小 (sm) → 中 (md) → 大 (lg) (各级倍率为 1.618)
```

### 8 像素网格概念 (8-Point Grid Concept)

```
所有的间距与尺寸均采用 8 的倍数：
├── 紧凑：4px (微调使用的半步)
├── 小型：8px
├── 中型：16px
├── 大型：24px, 32px
├── 特大：48px, 64px, 80px
└── 根据内容密度进行调整
```

### 关键尺寸原则 (Key Sizing Principles)

| 元素         | 考虑因素               |
| ------------ | ---------------------- |
| **触摸目标** | 最小舒适点击尺寸       |
| **按钮**     | 高度取决于重要层级     |
| **输入框**   | 高度应与按钮匹配以对齐 |
| **卡片**     | 内边距一致，保持呼吸感 |
| **阅读宽度** | 45-75 个字符为最佳     |

## 4. 配色原则 (Color Principles)

### 60-30-10 原则

```
60% → 主色/背景 (沉稳、中性的基调)
30% → 辅助色 (支撑区域)
10% → 强调色 (CTA、高亮、吸引注意)
```

### 色彩心理学 (用于辅助决策)

| 如果您需要…… | 考虑色相                    | 避免         |
| ------------ | --------------------------- | ------------ |
| 信任、冷静   | 蓝色系                      | 侵略性的红色 |
| 增长、自然   | 绿色系                      | 工业灰       |
| 活力、紧急   | 橙色、红色                  | 消极的蓝色   |
| 高端、创意   | 深青色 (Teal)、金色、祖母绿 | 廉价感的亮色 |
| 整洁、极简   | 中性色                      | 压倒性的色彩 |

### 选择流程

1. **行业是什么？** (缩小选择范围)
2. **情感基调是什么？** (挑选主色)
3. **亮色还是暗色模式？** (定下基调)
4. **询问老板** (如未指定)

详细色彩理论请参考：[color-system.md](color-system.md)

## 5. 排版原则 (Typography Principles)

### 比例尺选择 (Scale Selection)

| 内容类型  | 缩放比例  | 感受           |
| --------- | --------- | -------------- |
| 密集 UI   | 1.125-1.2 | 紧凑、高效     |
| 通用网页  | 1.25      | 平衡 (最常用)  |
| 社论/博客 | 1.333     | 易读、开阔     |
| Hero/展示 | 1.5-1.618 | 戏剧化的冲击力 |

### 配对概念 (Pairing Concept)

```
对比 + 和谐：
├── 差异足够大以便区分层级
├── 相似度足够高以保持凝聚力
└── 通常：展示字型 + 中性字型，或衬线 + 无衬线
```

### 易读性规则

- **行长度**：45-75 个字符为最佳。
- **行高**：正文建议采用 1.4-1.6。
- **对比度**：检查 WCAG 无障碍要求。
- **字号**：Web 端正文建议 16px 以上。

详细排版指南请参考：[typography-system.md](typography-system.md)

## 6. 视觉效果原则 (Visual Effects Principles)

### 玻璃拟态 (Glassmorphism) (适用时)

```
关键属性：
├── 半透明背景
├── 背景模糊 (Backdrop blur)
├── 用于界定边界的微弱边框
└── ⚠️ **警告：** 蓝/白背景的标准玻璃拟态已成为陈词滥调。要么激进地使用，要么干脆不用。
```

### 阴影层级 (Shadow Hierarchy)

```
高度概念：
├── 更高层的元素 = 更大的阴影
├── Y 轴偏移 > X 轴偏移 (光源来自上方)
├── 多层组合 = 更具真实感
└── 暗色模式：可能需要用发光 (Glow) 代替
```

### 渐变使用

```
和谐渐变：
├── 色轮上相邻的颜色 (类比色)
├── 或者同色相、不同亮度
├── 避免使用生硬的互补色对
├── 🚫 **禁止使用弥散/极光渐变** (漂浮的色块)
└── 不同项目间应保持极大的差异化
```

详尽的效果指南：[visual-effects.md](visual-effects.md)

## 7. 动画原则 (Animation Principles)

### 耗时概念 (Timing Concept)

```
时长基于：
├── 距离 (越远 = 越久)
├── 尺寸 (越大 = 越慢)
├── 重要性 (关键 = 清晰)
└── 环境 (紧急 = 快，高端 = 慢)
```

### 缓动函数选择 (Easing Selection)

| 动作 | 缓动函数    | 理由           |
| ---- | ----------- | -------------- |
| 进入 | Ease-out    | 减速，落定     |
| 离开 | Ease-in     | 加速，退出     |
| 强调 | Ease-in-out | 平滑，审慎     |
| 俏皮 | Bounce      | 有趣，充满活力 |

### 性能

- 仅对 `transform` 和 `opacity` 进行动画处理。
- 尊重用户的“减弱动态效果 (Reduced-motion)”偏好。
- 在低端设备上进行测试。

动画模式参考：[animation-guide.md](animation-guide.md)，进阶参考：[motion-graphics.md](motion-graphics.md)

## 8. “惊艳感”检查清单 ("Wow Factor" Checklist)

### 高级感指标 (Premium Indicators)

- [ ] **慷慨的空隙**：高端设计 = 呼吸空间。
- [ ] **细腻的深度感与维度感**。
- [ ] **平滑且有目的性的动画**。
- [ ] **细节至上**：对齐、一致性。
- [ ] **连贯的视觉节奏**。
- [ ] **自定义元素**：不仅使用默认设置。

### 信任构建 (Trust Builders)

- [ ] 在合适位置提供安全暗示。
- [ ] 社会认同 / 推荐语。
- [ ] 清晰的价值主张 (Value proposition)。
- [ ] 专业的意象图。
- [ ] 一致的设计语言。

### 情感触发 (Emotional Triggers)

- [ ] 能够唤起目标情感的 Hero 区域。
- [ ] 人文元素 (人脸、故事)。
- [ ] 进度/达成感指标。
- [ ] 令人愉悦的瞬间 (Moments of delight)。

## 9. 反模式 (应避免的行为) (Anti-Patterns)

### ❌ 偷懒设计的迹象

- 未经考虑便使用系统默认字体。
- 风格不符的库存素材图。
- 间距不统一。
- 同时使用过多竞争色。
- 一整面文字墙，缺乏层级感。
- 对比度不符合无障碍标准。

### ❌ AI 惯性模式 (必须避免！)

- **每个项目都用同样的颜色**。
- **默认采用“暗色 + 霓虹”风格**。
- **所有东西都用紫色/紫罗兰色 (紫色禁令 ✅)**。
- **为简单的落地页使用 Bento 网格**。
- **弥散渐变与发光效果**。
- **同样的布局结构 / Vercel 克隆版**。
- **不询问老板的偏好**。

### ❌ 黑暗模式 (不道德的设计)

- 隐藏费用。
- 虚假紧迫感。
- 强迫性操作。
- 具有欺骗性的 UI。
- 确认羞辱 (Confirmshaming)。

## 10. 决策流程总结 (Decision Process Summary)

```
针对每一个设计任务：

1. 约束条件 (CONSTRAINTS)
   └── 时间线、品牌、技术、受众分别是什么？
   └── 如不明确 → 询问

2. 内容 (CONTENT)
   └── 现有哪些内容？
   └── 层级如何划分？

3. 风格方向 (STYLE DIRECTION)
   └── 什么风格最适合当前上下文？
   └── 如不明确 → 询问 (不要直接采用默认值！)

4. 执行 (EXECUTION)
   └── 应用上述原则
   └── 对照反模式进行自查

5. 评审 (REVIEW)
   └── “这真的为用户服务吗？”
   └── “这与我的默认风格有区别吗？”
   └── “我会为此感到自豪吗？”
```

## 设计后工作流 (Post-Design Workflow)

在实现您的设计后，请运行审计：

```
1. 设计 (DESIGN)   → 阅读 frontend-design 原则 ← 您当前所在的阶段
2. 编码 (CODE)     → 实现设计方案
3. 审计 (AUDIT)    → 运行 web-design-guidelines 评审
4. 修复 (FIX)      → 解决审计发现的问题
```

> **下一步：** 编码完成后，使用 `web-design-guidelines` 技能来审计您的实现，包括无障碍性、焦点状态、动画以及性能问题。
