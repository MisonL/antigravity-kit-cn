# 上下文发现

> 在提出任何架构方案前，先收集上下文信息。

## 问题层级（先问用户）

1. **规模（Scale）**
   - 用户量级是多少？（10、1K、100K、1M+）
   - 数据规模是多少？（MB、GB、TB）
   - 事务速率是多少？（每秒/每分钟）

2. **团队（Team）**
   - 个人开发还是团队协作？
   - 团队规模与能力结构？
   - 是分布式团队还是同地办公？

3. **时间线（Timeline）**
   - 是 MVP（最小可行产品）/原型，还是长期产品？
   - 上线时效压力有多大？

4. **业务域（Domain）**
   - 以 CRUD（增删改查）为主，还是业务逻辑复杂？
   - 是否有实时性要求？
   - 是否有合规/监管要求？

5. **约束条件（Constraints）**
   - 是否有预算限制？
   - 是否要集成遗留系统？
   - 是否有技术栈偏好？

## 项目分类矩阵

```
                    MVP（最小可行产品）      SaaS（软件即服务）     Enterprise（企业级）
┌────────────────────────────────────────────────────────────────────────────┐
│ 规模           │ <1K                 │ 1K-100K          │ 100K+           │
│ 团队           │ 单人                │ 2-10             │ 10+             │
│ 时间线         │ 快（周级）           │ 中（月级）        │ 长（年级）        │
│ 架构           │ 简单架构             │ 模块化架构        │ 分布式架构        │
│ 模式           │ 最小模式集           │ 选择性模式        │ 全面模式体系      │
│ 示例           │ Next.js API（接口）  │ NestJS           │ Microservices（微服务）│
└────────────────────────────────────────────────────────────────────────────┘
```
